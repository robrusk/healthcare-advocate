HIPAA Compliance Analysis

Prepared: May 2026  •  For: Nonprofit Governance & Grant Applications
Summary Verdict HIPAA almost certainly does not apply to this application. Even so, the architecture is privacy-by-design: no data is stored, no accounts are created, and no third-party trackers are present. The app would satisfy HIPAA's spirit regardless of whether it is legally required to.

1. Does HIPAA Apply?

HIPAA (the Health Insurance Portability and Accountability Act) regulates two categories of organizations:

  • Covered entities — health plans, healthcare providers, and healthcare clearinghouses
  • Business associates — vendors that process protected health information (PHI) on behalf of a covered entity

Healthcare Advocate is neither. It is a patient advocacy tool — it helps patients manage their own health information. Under HIPAA, a patient may share their own PHI with any person or organization of their choosing. That disclosure is not regulated.

Category Applies? Reason
Covered entity (health plan) No Not a health plan
Covered entity (healthcare provider) No Not providing medical services
Covered entity (clearinghouse) No Not processing billing transactions on behalf of providers
Business associate No Not handling PHI on behalf of any covered entity — patients share their own data directly

Conclusion: The organization is not subject to HIPAA's Privacy Rule or Security Rule. No Business Associate Agreements are required with users.

2. Privacy-by-Design Architecture

Although HIPAA does not apply, the application was architected to handle health data with the same care a covered entity would apply. The following technical controls are in place:

Data Flow — Nothing Is Stored

User's browser
  ↓ base64-encode document in memory
Cloudflare Worker (stateless proxy)
  ↓ forward request — no logging, no storage
Anthropic Claude API
  ↓ AI analysis — response returned
User's browser
  ↓ display results — React state only
Tab closed → all data gone

No User Accounts or Persistent Storage

The application has no login, no user profiles, and no database. Uploaded documents and analysis results live only in browser memory (React state) for the duration of the session. When the tab is closed, all data is gone. Nothing is written to localStorage, IndexedDB, or any server-side store.

No Third-Party Trackers

There is no Google Analytics, Facebook pixel, advertising network, or any other third-party script that could receive health data as a side effect of normal use. The only external service contacted is the Anthropic Claude API, accessed through the organization's own Cloudflare Worker proxy.

Encrypted in Transit

All data travels over HTTPS. Cloudflare handles TLS termination for both the application and the Worker proxy. Data is encrypted end-to-end between the user's browser and the Anthropic API.

Anthropic API Data Handling

Anthropic's API (as distinguished from its consumer products) does not train models on API inputs by default, and inputs are not retained for product improvement. Per Anthropic's API terms, inputs may be retained for a short period (typically up to 30 days) for automated abuse monitoring — this review is automated, not human, unless a specific input is flagged. Zero Data Retention (ZDR) is available under Anthropic's Business/Enterprise agreement and would eliminate even this window; worth requesting when pursuing API credits as a nonprofit.

3. Cloudflare Worker Proxy

All Claude API calls are routed through a Cloudflare Worker rather than directly from the browser. This was done for API key security (the key never appears in the browser bundle), but it has an additional privacy benefit: the Worker is a stateless passthrough and does not store document content.

The Worker enforces CORS to accept requests only from authorized origins (healthcareadvocate.org and the local development server). It adds the required Anthropic headers and forwards the request. No request body logging exists in the Worker code.

Note on Cloudflare infrastructure logging: Like all CDN-hosted sites, Cloudflare records standard request metadata — IP address, timestamp, URL, and HTTP response code — for some period as part of normal hosting operations. Cloudflare does not log request bodies (the document content) at the edge. The privacy statement should be precise: we do not log document contents; standard hosting metadata (IP, timestamp, URL) is collected by Cloudflare as it is by any web host.

Action item before public launch: The Worker source code was deployed directly to Cloudflare and does not currently live in the project repository. Adding it to the repo (e.g., cloudflare/worker.js) makes the no-logging claim independently verifiable by anyone reviewing the nonprofit's code.

4. Service Worker and Browser Cache

The application is a Progressive Web App (PWA) and uses a service worker. The service worker caches only the application shell — HTML, JavaScript, and CSS files needed to run the app offline. It does not cache user-uploaded documents, API responses, or any PHI. This behavior is controlled by the Vite PWA plugin configuration and has been verified.

5. Considerations and Recommendations

Anthropic Business Associate Agreement

Anthropic offers a HIPAA Business Associate Agreement (BAA) to enterprise customers. A BAA would only be required if HIPAA applied to the organization — which it likely does not. However, if the organization's legal counsel determines that a more conservative posture is warranted, Anthropic's enterprise agreement provides this option.

Recommended: Home Screen Privacy Statement

Action Item Before public launch, add a one-sentence privacy statement visible on the home screen:

"We don't store your documents or personal information. Nothing leaves your device except the document you upload for analysis."

This is factually accurate and builds trust with users who may be apprehensive about uploading health documents.

FTC Act and State Privacy Laws

Even where HIPAA does not apply, the FTC Act prohibits unfair or deceptive data practices. California's Confidentiality of Medical Information Act (CMIA) and similar state laws may impose obligations on entities that handle medical information. The privacy-by-design architecture described in Section 2 — no storage, no accounts, no trackers — provides strong protection against claims under these frameworks as well, because there is no data to misuse.
